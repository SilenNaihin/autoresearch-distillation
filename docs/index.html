Abstract

We train Qwen3-14B to optimize a GPT pretraining script through Self-Distillation Policy Optimization (SDPO), where the model edits code via bash tools, experiments run on H100 GPUs, and scalar rewards flow back into training. On Karpathy's autoresearch benchmark, SDPO achieves val_bpb = 1.028 — a 2.7% improvement over the unmodified baseline, comparable to Claude's 2.8% in the original experiments. We compare against two baselines — multi-turn in-context learning and single-turn random exploration — finding that feedback adds consistency but not better outcomes, while weight updates are what break past the sampling distribution.

Three Approaches

SDPO RL

The model generates rollouts — editing train.py via bash tool calls — and experiments execute on remote H100s. Rewards (how much val_bpb improved) flow back through GRPO updates. The teacher is the model itself: successful rollouts become self-distillation targets, and environment feedback (diffs, metrics, crash info) is injected into the teacher prompt. Weight updates encode exploration patterns that compound across training — infinite effective memory, unlike context-bounded approaches.

ICL Baseline In-Context

Same model, same tools, no weight updates. The model accumulates experiment feedback in its context window across 50 sequential turns: top results with full diffs, classified crash reports, and structured history. All learning happens through context conditioning. Best result: val_bpb = 1.038 across multiple runs.

Single-turn Baseline No Feedback

N independent single-shot calls with no history and no feedback. Just “here’s train.py, improve it” repeated N times, taking the best result. This measures the entropy of the sampling distribution — what you get from pure random exploration at temperature 1.0. At 50 turns, single-turn achieves val_bpb = 1.032 — actually beating ICL’s best of 1.038, suggesting the feedback loop adds consistency but not fundamentally better outcomes at this scale.

What the Model Finds

The dominant insight across both SDPO and ICL: trade model size for training throughput. Within the fixed 5-minute budget, a smaller model that completes more gradient steps outperforms a larger model that trains fewer steps. The best SDPO result reduces ASPECT_RATIO from 64 to 43, shrinking the model from ~50M to ~33M parameters but nearly doubling training steps (481→952).

Learned intuition through weight updates

The SDPO prompt contains zero experiment history — just the system prompt and train.py. Yet after training, the model generates chain-of-thought that references “previous attempts” and their outcomes. These aren’t memories from context; they’re learned intuitions encoded in the weights through SDPO, surfacing as hallucinated but directionally correct recollections:

<think>

I need to modify the train.py script to lower the val_bpb. The user tried increasing the DEPTH from 8 to 12 before, but that caused an increase in val_bpb. The feedback from the previous attempt shows that when DEPTH was increased to 12, the val_bpb went up to 1.06 — so that was a bad idea.

The user’s previous attempt was to change DEPTH from 8 to 12, but that didn’t help. The previous attempt’s feedback shows that increasing depth caused a higher val_bpb. So that’s not the way.

Alternatively, maybe I should look at modifying other parameters. The user’s feedback from an earlier attempt shows that changing ASPECT_RATIO — the model’s dimensions are computed based on ASPECT_RATIO and DEPTH. The head dimension must be a power of two for FlashAttention 3…

…looking at the activation function: in the MLP, the current code uses F.relu(x).square(). Maybe using a different activation function, like GELU, could help. But in the previous attempt, changing the model architecture didn’t help. So perhaps a different approach is needed.

The model has never seen experiment feedback in its prompt. There is no “user” and no “previous attempt.” But through SDPO training on hundreds of rollouts, the model’s weights have internalized that increasing depth alone hurts val_bpb, that ASPECT_RATIO matters, and that architectural changes need careful consideration — all encoded as intuition, not context.

Weights vs. Context

The core question: does RL training add value beyond what in-context learning provides?

Single-turn beats ICL. The single-turn baseline (1.032) actually outperforms ICL (1.038) through a single lucky hit. The feedback loop gives ICL more consistent improvements, but doesn’t produce fundamentally better outcomes — random exploration at temperature 1.0 finds better configurations by chance. Context conditioning adds reliability, not capability.

SDPO breaks out. Weight updates encode exploration patterns that persist across rollouts without consuming context. The model learns which kinds of changes tend to work — a form of meta-learning that context conditioning can’t replicate. SDPO achieves 1.028, a clear separation from both baselines, with its best result at step 56 of 60 suggesting the policy is still improving.

The gap between single-turn and ICL measures what context contributes (nothing — single-turn actually wins). The gap between both baselines and SDPO measures what weights contribute.

Comparison to Claude

Karpathy’s original experiments used Claude on H100 GPUs, running 126 experiments over multiple cycles. The best result improved val_bpb by 2.8% (0.998 → 0.970). Key improvements: batch halving for more gradient steps (−0.012), adding depth (−0.004), embedding LR tuning (−0.003), and weight decay adjustments.

Our SDPO result (−2.7%) nearly matches this despite using a 14B open-source model vs. a frontier model. The dominant insight is the same across both: throughput over parameters — trade model size for more training steps within the fixed budget. Both Claude and Qwen3-14B independently converge on this strategy.

Built On

Autoresearch (Karpathy) — A single-file GPT pretraining script with Muon+Adam optimization, FlashAttention 3, and sliding window attention. The benchmark: modify train.py to minimize val_bpb within a 5-minute training budget on a single H100.

SDPO (Qu et al., 2025) — Self-Distillation Policy Optimization. Uses the model’s own successful rollouts as teacher targets, avoiding the need for a separate reward model. We extend it with agentic tool use and live environment rewards from real GPU experiments.
