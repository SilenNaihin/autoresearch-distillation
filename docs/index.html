Abstract

Can a small open-source model learn to do ML research through reinforcement learning on live experiment outcomes? We take Karpathy's autoresearch benchmark — a GPT pretraining script optimized for val_bpb (bits per byte) under a 5-minute training budget — and train Qwen3-14B to propose code modifications using Self-Distillation Policy Optimization (SDPO). The model edits train.py via bash tools, experiments run on H100 GPUs, and scalar rewards flow back into training. No separate reward model or offline data — the agent trains on live experiment outcomes.

SDPO achieves val_bpb = 1.029, a 2.6% improvement over the unmodified baseline (1.056), outperforming both a strong multi-turn in-context learning baseline and matching the improvements found by frontier models in the original autoresearch experiments.

How It Works

The agent receives train.py (a ~500-line GPT pretraining script using Muon+Adam optimizer, FlashAttention 3, and sliding window attention) and must propose modifications to lower validation loss. Each iteration is an edit–experiment cycle: 

   Agent (Qwen3-14B)                              GPU Fleet (H100s)
  ┌──────────────────────┐                        ┌──────────────────┐
  │                      │                        │                  │
  │  1. Read train.py    │                        │                  │
  │  2. Think step by    │    modified train.py   │  4. Execute      │
  │     step in <think>  │  ────────────────────> │     uv run       │
  │  3. Edit via sed     │                        │     train.py     │
  │                      │    val_bpb + metrics   │     (5 min)      │
  │  6. Receive reward   │  <──────────────────── │                  │
  │     + feedback       │                        │  5. Parse output │
  │                      │                        │                  │
  └──────────────────────┘                        └──────────────────┘
           │
           │  reward = max(0, 1.056 - val_bpb)
           ▼
  ┌──────────────────────┐
  │  SDPO Weight Update  │  (or context feedback for ICL baseline)
  └──────────────────────┘

The model uses a bash tool to make targeted sed edits. It does not run the experiment itself — the modified script is dispatched to a remote GPU, and the outcome feeds into the next iteration. Chain-of-thought is enabled: the model reasons inside <think> blocks before acting.

Methods

We compare three approaches for learning to optimize the training script, all using the same Qwen3-14B base model:

SDPO RL Training

Self-Distillation Policy Optimization with live experiment rewards. The model generates rollouts (editing train.py via bash), experiments execute on 5 parallel H100s, and rewards flow back through GRPO updates. The teacher is the model itself — successful rollouts become self-distillation targets. Environment feedback (diffs, metrics, crash info) is injected into the teacher prompt. A novelty signal encourages exploration: novel diffs receive a bonus, while repeated diffs are flagged.

batch size 16 · 100 epochs · 5 experiment GPUs

ICL Baseline In-Context Learning

Same model, same agent interface, same tools — no weight updates. The model accumulates experiment feedback in its context window across 30 sequential turns. Each turn, it sees the original train.py plus a structured history: top-3 results with full diffs, one-line summaries for the rest, and classified crash reports. Learning happens purely through in-context conditioning.

30 sequential turns · 30 tool calls/turn · frozen weights
MCMC Power Sampling Ablation

Based on "Reasoning with Sampling" (Karan & Du, 2025). The hypothesis: RL post-training primarily sharpens the base model distribution. If so, we can approximate the same effect at inference time by sampling from pα using block-wise MCMC — no training required.

Uses single-shot generation (no multi-turn tool use) since MCMC requires an unbroken autoregressive chain. The model generates reasoning + sed commands in one pass, which are parsed and applied. This introduces a confound: if it underperforms, it may be the single-shot format rather than the sampling method.

Parameters: α=4.0, 16 blocks, 10 MCMC steps/block, temperature=1.0 (proposal = base model).

Example: What the Model Learns to Try

Early in training, the model makes conservative hyperparameter tweaks. As training progresses, it explores increasingly creative modifications — architecture changes, activation function swaps, optimizer internals:

--- a/train.py +++ b/train.py @@ -103,7 +103,7 @@ def forward(self, x): x = self.c_fc(x) - x = F.relu(x).square() + x = F.gelu(x) x = self.c_proj(x) return x

The model discovers that swapping the ReluSquared activation for GELU in the MLP — one of many creative diffs generated during SDPO training.

Observations

Prompt Sensitivity

Both methods are remarkably sensitive to prompt engineering. Small changes to system prompts — adding hardware constraints, tweaking exploration encouragement, restructuring feedback format — can cause the model to collapse into degenerate states or unlock new exploration. This sensitivity appears more pronounced at the 14B scale; larger models like Claude show less fragility to prompt phrasing.

Novelty Feedback Matters

A key stabilization technique: tracking seen diffs via a hash set and providing explicit novelty feedback. Novel changes receive encouragement ("Continue to try new creative attempts"), while repeated diffs are redirected ("This exact set of changes has been tried before. Try a different approach"). Without this, the model converges to repeating the same narrow set of modifications.

In-Context Learning Has a Ceiling

The ICL baseline plateaus after ~15–20 turns. The model exhausts novel ideas within the same narrow exploration space: tuning the same ~6 hyperparameters (depth, aspect ratio, learning rates, batch size) within small ranges. It rarely touches optimizer internals (Muon momentum, beta2) or attempts architectural innovations — areas where the best results come from. SDPO's weight updates appear to encode exploration patterns that pure context conditioning cannot.

Crash Rate as a Signal

~40% of experiment attempts crash (OOM, FlashAttention errors, assertion failures). Adding hardware constraints to the system prompt (VRAM limits, head_dim requirements) reduced wasted turns but didn't eliminate them. The crash rate stays roughly flat across turns for ICL, suggesting the model doesn't effectively learn to avoid failure modes from context feedback alone.

VERL (via the SDPO fork) orchestrates RL training: rollout generation, advantage estimation, and policy updates. Our custom AutoresearchAgentLoop extends VERL's ToolAgentLoop with persistent bash tool instances, experiment dispatch, and reward computation. Tool response tokens are masked (response_mask=0) so the model only trains on its own decisions.

A shared persistent cache (keyed by diff SHA256) avoids re-running identical experiments across both SDPO and baseline. OOM crashes are cached deterministically; transient failures are retried. Within SDPO training steps, cache hits are filtered by step number to prevent ordering-dependent novelty bonuses across parallel workers.

Built On

Autoresearch (Karpathy) — The GPT pretraining benchmark. A single-file train.py that trains a GPT on FineWeb-Edu with Muon+Adam optimization, FlashAttention 3, and sliding window attention. The target metric val_bpb (bits per byte) measures compression quality on held-out data.

SDPO (Qu et al., 2025) — Self-Distillation Policy Optimization. An RL method built on VERL that uses the model's own successful rollouts as teacher targets, avoiding the need for a separate reward model. We extend it with agentic tool use and live environment rewards.

Reasoning with Sampling (Karan & Du, 2025) — The theoretical basis for our MCMC power sampling ablation. Shows that RL post-training primarily sharpens the base model distribution, suggesting inference-time MCMC as an alternative.
